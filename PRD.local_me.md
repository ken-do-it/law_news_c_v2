## 1. 제품 개요

### 1.1 제품명
**LawNGood News Analyzer** — AI 기반 소송금융 투자 케이스 자동 발굴 시스템

### 1.2 배경 및 목적

소송금융(Litigation Finance)은 피해자가 소송 비용 부담 없이 사건을 진행할 수 있도록
투자자가 비용을 대신 지불하고 승소 시 수익을 공유하는 금융 서비스입니다.

**현재의 문제**:
- 담당자가 매일 수십 개 뉴스 매체를 수동으로 검색
- 관련 기사를 일일이 읽고 투자 가치를 주관적으로 판단
- 시간 소모가 극심하며 인간적 한계로 인한 케이스 누락 위험 상존

**LawNGood의 해결책**:
- 네이버 뉴스 API로 관련 기사를 자동 수집 (60분 주기)
- Google Gemini AI가 6개 객관적 기준으로 소송금융 투자 적합성을 자동 판단
- 유사 사건을 자동으로 그룹핑하여 트렌드 파악

### 1.3 타겟 사용자

| 사용자 | 주요 사용 목적 |
|--------|----------------|
| 소송금융 투자사 심사팀 | 케이스 발굴, 일일 리뷰 |
| 법무팀 | 사건 동향 모니터링 |
| 경영진 | 투자 현황 대시보드 조회 |

### 1.4 핵심 가치 제안

| Before (수동) | After (LawNGood) |
|---------------|------------------|
| 1인 기준 하루 2~3시간 뉴스 수동 검색 | 자동 수집 (60분 주기) |
| 담당자 경험에 의존한 케이스 선별 | 6개 객관적 기준에 의한 AI 판단 |
| 유사 사건 중복 검토 | 자동 케이스 그룹핑 |
| 결과 정리 엑셀 수작업 | 원클릭 엑셀 내보내기 |

## 2. 시스템 아키텍처 개요

```
[네이버 뉴스 API]
        ↓ 키워드 검색 (60분 주기)
[Django Backend]
  ├── 기사 수집 (BeautifulSoup 본문 추출)
  ├── 중복 제거 (URL unique constraint)
  ├── Gemini 2.5 Flash API 분석
  └── 자동 케이스 그룹핑 (유사도 0.6 임계값)
        ↓ REST API (/api/*)
[React + TypeScript Frontend]
  ├── 대시보드 (KPI 차트)
  ├── 분석 목록 (필터/검색)
  ├── 분석 상세 (케이스 심층 분석)
  └── 설정 (키워드 관리)
```

**기술 스택**:

| 계층 | 기술 |
|------|------|
| Backend | Django 5.x + Django REST Framework + APScheduler |
| AI | Google Gemini 2.5 Flash |
| Database | SQLite (dev) / PostgreSQL (prod 권장) |
| Frontend | React 19 + TypeScript + Tailwind CSS 4 + Recharts |
| Build | Vite 7 |

## 3. 핵심 기능 요구사항

### 3.1 자동 뉴스 수집 `P0`

- 서버 실행 30초 후 최초 1회 자동 실행, 이후 설정 주기(기본 60분)마다 반복
- 동시 실행 방지 (동일 파이프라인 중복 실행 금지)
- 기사 URL 기준 중복 저장 방지
- 소스: 네이버 뉴스 검색 API (키워드당 최대 100건)
- 키워드: 관리자 UI에서 동적 설정 (기본: 소송, 손해배상, 집단소송 등)
- 수집 데이터: 기사 제목, 본문, URL, 작성자, 게재일, 언론사 (80개 이상 사전 매핑)

### 3.2 AI 투자 적합성 분석 `P0`

- 기사 수집 즉시 자동 분석 트리거
- 분석 실패 시 자동 재시도 (최대 2회, 총 3회 시도)
- UI에서 개별 기사 재분석 수동 트리거 지원

**6개 투자 적합성 판단 기준**:

| 조건 | 내용 |
|------|------|
| C1 | 상대방의 명확한 책임 존재 |
| C2 | 상대방의 충분한 자력 (대기업/공기관 등) |
| C3 | 집단적 피해 (50명 이상) |
| C4 | 대규모 피해 (10억 원↑ 또는 피해자 1만명↑) |
| C5 | 증거 존재 또는 확보 가능 |
| C6 | 공개 절차 진행 중 (수사/감사 등) |
| X1 | 결격 사유: 사건 종결 (확정판결/합의 완료) |

**평가 등급**:
- **High**: 4개 이상 충족 + X1 없음
- **Medium**: 2~3개 충족 + X1 없음
- **Low**: 1개 이하 또는 X1 해당

**AI 추출 정보**: 투자 적합도, 판단 근거, 사건 유형, 피고, 피해 금액, 피해자 수, 소송 진행 단계, 단계 상세, AI 요약, 법적 관련성 여부

### 3.3 케이스 자동 그룹핑 `P1`

- 동일 사건 관련 기사를 자동으로 하나의 케이스로 묶음
- 유사도 알고리즘: 토큰 기반 문자열 유사도 (임계값 0.6)
- 케이스 ID 자동 부여 (예: `CASE-2026-001`, 연도별 순차 번호)
- 연관 기사 수 실시간 집계 및 표시

### 3.4 대시보드 `P0`

**KPI 카드** (6개):
- 오늘 수집 건수, 분석 대기, 오늘 분석 완료, 전체 분석 완료, 오늘 High 건수, 오늘 Medium 건수

**차트** (3개):
- 적합도 분포 파이차트 (High/Medium/Low 비율)
- 사건 유형별 상위 10개 막대 차트
- 최근 7일 분석 트렌드 라인차트 (전체/High/Medium)

**최신 분석 테이블**: 최근 5건 요약 표시

### 3.5 분석 목록 `P0`

**필터**:
- 텍스트 검색 (제목, 피고, 사건명)
- 적합도 필터 (High/Medium/Low 복수 선택 가능)
- 소송 단계 필터 (5단계)
- 케이스별 그룹핑 토글 (기본: ON)
- 법적 관련 없는 기사 포함 토글 (기본: OFF)

**표시 컬럼**: 적합도, 제목(+연관 건수 뱃지), 사건유형, 피해자수, 피고, 피해금액, 소송단계, 케이스ID, 날짜

**출력**: 20건/페이지 페이지네이션, 엑셀 내보내기 (현재 필터 상태 반영)

### 3.6 분석 상세 `P1`

**좌측**: 기사 원문 정보, AI 3~5문장 요약, 투자 적합도 판단 근거, 연관 기사 목록
**우측 (고정)**: 케이스 핵심 정보 카드, 재분석 버튼, 엑셀 내보내기 버튼, LLM 메타데이터

### 3.7 키워드 설정 `P1`

- 현재 활성 키워드 태그 표시
- 신규 키워드 추가 (즉시 반영)
- 키워드 삭제 (다음 수집 주기부터 미적용)

### 3.8 엑셀 내보내기 `P1`

- 15개 컬럼 (번호, 케이스ID, 제목, 언론사, 날짜, 적합도, 판단근거, 유형, 피고, 피해금액, 피해자수, 단계, 단계상세, AI요약, 원문링크)
- 적합도 셀 색상 코딩, 헤더 스타일링, 행 고정
- 현재 필터 조건 유지한 상태로 내보내기

## 4. 비기능 요구사항

### 4.1 성능
- API 응답 시간: 목록 조회 < 500ms
- 엑셀 생성 시간: < 5초 (500건 기준)
- 동시 파이프라인 실행: 최대 1개

### 4.2 신뢰성
- LLM 분석 실패 시 자동 재시도 (최대 2회)
- 최대 재시도 초과 시 `failed` 상태로 보존 (데이터 손실 없음)
- 크롤링 실패 시 개별 기사 스킵 후 계속 진행

### 4.3 확장성
- 키워드 수 제한 없음
- 언론사 목록 동적 추가 (Admin UI)
- Celery로 전환 가능한 동기 파이프라인 구조 (현재: APScheduler)

### 4.4 보안
- API 키 환경변수 관리 (`.env`)
- CORS 화이트리스트 제한
- Django Admin 접근 제한

## 5. 데이터 흐름

```
[수집 단계]
APScheduler 트리거 (60분 주기)
→ 활성 키워드별 네이버 API 호출
→ 기사 URL 중복 확인
→ BeautifulSoup 본문 추출
→ Article 저장 (status: pending)

[분석 단계]
pending 기사 전체 조회
→ 기사별 Gemini 2.5 Flash API 호출 (JSON mode)
→ 응답 파싱 및 필드 검증
→ 유사 케이스 그룹 매칭 (기존 케이스명과 유사도 비교)
→ Analysis 저장
→ Article status → analyzed (실패 시 → failed)

[조회 단계]
Frontend API 요청 (with filter params)
→ Django QuerySet 필터링 + 페이지네이션
→ Serializer 직렬화
→ Paginated JSON 응답
```
