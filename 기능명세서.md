## 1. articles 앱 — 뉴스 수집

> 경로: `backend/articles/`
>
> 역할: 네이버 뉴스 API를 통한 기사 수집, 키워드/언론사 관리

### 1-1. 모델 (models.py)

#### MediaSource (언론사)

| 필드 | 타입 | 설명 |
|------|------|------|
| `id` | BigAutoField (PK) | 자동 증가 기본키 |
| `name` | CharField(100), unique | 언론사명 (예: "조선일보") |
| `url` | URLField(500), blank | 언론사 홈페이지 URL |
| `is_active` | BooleanField, default=True | 활성화 여부 |
| `created_at` | DateTimeField, auto_now_add | 등록일시 |

- 초기 데이터: 80개 언론사 (종합일간지, 방송/통신, 경제, 인터넷, IT 전문, 매거진, 지역)

#### Keyword (검색 키워드)

| 필드 | 타입 | 설명 |
|------|------|------|
| `id` | BigAutoField (PK) | 자동 증가 기본키 |
| `word` | CharField(50), unique | 검색 키워드 (예: "집단소송") |
| `is_active` | BooleanField, default=True | 활성화 여부 |
| `created_at` | DateTimeField, auto_now_add | 등록일시 |

- 초기 데이터: 소송, 손해배상, 집단소송, 공동소송, 피해자, 피해보상, 피해구제 (7개)

#### Article (기사)

| 필드 | 타입 | 설명 |
|------|------|------|
| `id` | BigAutoField (PK) | 자동 증가 기본키 |
| `source` | FK → MediaSource, nullable | 언론사 |
| `title` | CharField(500) | 기사 제목 (HTML 태그 제거됨) |
| `content` | TextField | 기사 본문 (네이버 뉴스 페이지에서 추출) |

| `published_at` | DateTimeField | 기사 발행일시 |
| `collected_at` | DateTimeField, auto_now_add | 수집일시 |

### 1-3. 크롤러 (crawlers.py)

| 함수 | 설명 |
|------|------|
| `search_naver_news(keyword, display=100)` | 네이버 뉴스 검색 API 호출. 키워드별 최대 100건 반환 |
| `fetch_article_content(naver_url)` | 네이버 뉴스 페이지에서 `#dic_area` 영역의 본문 텍스트 추출 |
| `clean_html(text)` | HTML 태그 및 엔티티 제거 |
| `parse_naver_date(date_str)` | 네이버 API 날짜 형식 → datetime 변환 |
| `extract_source_from_url(url)` | URL 도메인을 언론사명으로 매핑 (60개+ 도메인) |
| `extract_source_from_naver_page(url)` | 네이버 뉴스 페이지에서 언론사 로고 alt 텍스트 추출 |

**언론사 식별 2단계 프로세스:**
1. `extract_source_from_url`: originallink 도메인으로 매핑 (빠르고 정확)
2. `extract_source_from_naver_page`: 1단계 실패 시 네이버 페이지 스크래핑 (느리지만 범용)

### 1-4. 태스크 (tasks.py)

#### `crawl_news()` — Celery 태스크

- 활성화된 Keyword를 순회하며 `search_naver_news()` 호출
- 검색 결과에서 URL 중복 체크 후 신규 기사만 저장
- 언론사 식별: `_resolve_source()` (URL 매핑 → 네이버 스크래핑)
- 저장 시 `status="pending"` 설정
- 완료 후 `analyze_pending_articles.delay()` 자동 트리거 (Redis 필요)

#### `crawl_news_sync()` — 동기 실행

- `crawl_news()`를 Celery 없이 직접 호출하는 래퍼
- `just crawl` 명령에서 사용

### 1-5. 관리 명령어 (management/commands/)

#### `seed_initial_data`

- 7개 기본 키워드 생성 (`get_or_create`)
- 80개 언론사 생성 (종합/방송/경제/인터넷/IT/매거진/전문지/지역)
- `just seed` 명령에서 사용

> 경로: `backend/analyses/`
>
> 역할: LLM 기반 소송금융 적합도 분석, 사건 그룹핑, 엑셀 내보내기

### 2-1. 모델 (models.py)

**클래스 메서드:**

### 2-3. LLM 프롬프트 (prompts.py)

#### SYSTEM_PROMPT

- AI 역할: "소송금융 투자를 검토하는 전문 심사역"
- 이중 관점: 원칙적 법률 전문가 + 공격적 비즈니스 전략가
- 판단 기준 C1~C6, X1 상세 설명
- JSON 응답 형식 지정 (12개 필드)
- `is_relevant` 판단 기준 상세 (true/false 조건 나열)

#### FEW_SHOT_EXAMPLES (3건)

| 예시 | 판정 | 핵심 |
|------|------|------|
| 쿠팡 개인정보 유출 + 소상공인 집단소송 | High | C1, C2, C3, C6 충족 |
| 압타밀 분유 리콜 | Medium | C2, C3 충족 |
| 5·18 유족 국가배상 판결 확정 | Low | X1 종결 사건 |

#### `build_messages(title, content, existing_case_names)`

- 본문 3000자 제한 (토큰 절약)
- 기존 사건 그룹 목록을 시스템 프롬프트 말미에 추가
- system → few-shot (user/assistant 3쌍) → user(실제 기사) 순서로 메시지 구성

### 2-4. LLM 응답 검증 (validators.py)

#### `validate_and_parse(raw_response)`

**처리 흐름:**

1. JSON 파싱 (`json.loads`)
2. 필수 필드 검증: `suitability`, `suitability_reason`, `case_category`, `summary`
3. `suitability` 값 검증 (High/Medium/Low 아니면 Low로 보정)
4. `stage` 값 검증 (유효하지 않으면 빈값으로 보정)
5. 기본값 설정: defendant, damage_amount, victim_count, stage, stage_detail, case_name, is_relevant
6. `is_relevant` 타입 검증 (boolean 아니면 True로 보정)
7. `summary` 길이 제한 (1000자)

### 2-5. 태스크 (tasks.py)

#### LLM 호출 함수

| 함수 | 설명 |
|------|------|
| `call_gemini(messages)` | google-genai SDK로 Gemini API 호출 (기본 LLM) |
| `call_openai(messages)` | OpenAI SDK로 GPT-4o API 호출 (폴백 LLM) |

반환값: `(응답텍스트, prompt_tokens, completion_tokens)`

#### 사건 그룹 관련 함수

| 함수 | 설명 |
|------|------|
| `get_existing_case_names()` | 활성 사건 그룹 이름 목록 조회 (is_relevant=True 기사가 있는 그룹만) |
| `_case_similarity(a, b)` | 핵심 엔티티 기반 사건명 유사도 계산 (stopword 제외) |
| `find_or_create_case_group(case_name)` | 1) 정확 일치 → 2) 유사도 매칭(≥0.6) → 3) 신규 생성 |

**유사도 매칭 세부:**

- 기본: `SequenceMatcher` 문자열 유사도
- 법률 일반 용어 28개 stopword 제외 (소송, 분쟁, 사건, 피해 등)
- 핵심 엔티티(2자 이상) 공유 시 보너스 +0.25
- 3자 이상 엔티티의 부분 문자열 매칭 시 보너스 +0.25
- 임계값: 0.6 이상이면 기존 그룹에 매칭

#### `analyze_single_article(article)` — 핵심 분석 로직

**처리 흐름:**

1. `article.status = "analyzing"` 저장
2. `get_existing_case_names()`로 기존 사건 목록 조회
3. `build_messages()`로 프롬프트 구성
4. LLM 우선순위 결정: Gemini(기본) → OpenAI(폴백)
5. Primary LLM 호출 → 실패 시 Fallback 호출
6. `validate_and_parse()`로 응답 검증
7. `find_or_create_case_group()`으로 사건 그룹 연결
8. Analysis 레코드 생성 (실제 사용 모델명 기록)
9. `article.status = "analyzed"` 저장

#### `analyze_pending_articles()` — Celery 태스크

- `status="pending"` 기사를 `collected_at` 순으로 일괄 처리
- 각 기사마다 `analyze_single_article()` 호출
- 결과 반환: `{"total": N, "success": N, "failed": N}`

#### `reanalyze_article(article_id)` — Celery 태스크

- 기존 Analysis 삭제 → status를 "pending"으로 복구 → 재분석

### 2-6. 엑셀 내보내기 (export.py)

#### `export_analyses_to_excel(queryset)`

- openpyxl로 스타일링된 .xlsx 파일 생성
- 헤더 행: 배경색, 볼드, 고정
