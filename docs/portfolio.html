Key Features

핵심 기능

단순 뉴스 수집을 넘어, 업무 플로우 전체를 자동화했습니다.

1

자동 뉴스 수집 파이프라인

네이버 뉴스 API로 7개 법률 키워드(소송, 손해배상, 집단소송 등)를 60분마다 자동 검색합니다. URL 기반 중복 제거 후 기사 본문을 직접 스크래핑하며, 95개 언론사 도메인 매핑으로 언론사명을 정확히 식별합니다.

2

LLM 기반 소송금융 적합도 판정

Gemini 2.5 Flash에게 소송금융 심사역 역할을 부여하고, 6가지 투자 기준(C1~C6)과 Few-shot 예시 3건을 함께 전달합니다. 응답은 JSON 형식으로 강제하여 Pydantic으로 검증 후 저장합니다. Gemini 실패 시 OpenAI GPT-4o로 자동 폴백합니다.

3

사건 자동 그룹핑 알고리즘 (Case ID)

같은 사건을 다룬 기사들을 하나의 Case ID로 자동 묶습니다. 3단계 매칭 — ① 정확 이름 일치, ② Python SequenceMatcher 유사도 ≥ 0.85, ③ 제목 핵심 키워드 4개 이상 공유 — 을 순서대로 적용합니다. 법률 불용어(소송, 피해 등)를 제거하여 "쿠팡 개인정보 유출"과 "해외직구 소비자 피해"가 잘못 묶이는 문제를 방지합니다.

4

대시보드 & 분석 목록 UI

케이스 단위 뷰에서 사건별 적합도 분포(High/Medium/Low 기사 수)를 한눈에 보여줍니다. 분석일, 피해 규모, 적합도 기준 정렬과 발행일 기준 정렬을 지원하며, 케이스 상세 페이지에서 소속 기사를 펼쳐보고 로앤굿 심사 결과를 직접 입력할 수 있습니다.

5

엑셀 내보내기 & 운영 관리

현재 필터 조건을 그대로 엑셀로 내보낼 수 있습니다. just analyze --limit N으로 건수를 제어하며 bulk 재분석이 가능하고, just db-reset-analyses로 기사를 보존한 채 분석 결과만 초기화할 수 있습니다.

Engineering Challenges

기술적 도전과 해결

개발 과정에서 맞닥뜨린 실제 문제들과 해결 방식입니다.

문제 1

SQLite 동시 쓰기 잠금

APScheduler가 분석 태스크를 실행하는 동안 Django 개발 서버의 다른 요청이 DB에 쓰려 하면 OperationalError: database is locked가 발생했습니다. SQLite의 단일 쓰기 제약이 근본 원인이었습니다.

해결

SQLite에서 PostgreSQL 16 (Docker)으로 마이그레이션했습니다. Django의 DATABASE_URL 파싱 로직을 settings.py에 직접 구현하여 추가 라이브러리 없이 전환했습니다. 데이터는 Python serializers.serialize/deserialize로 마이그레이션했습니다.

문제 2

AI의 비결정적 사건명 생성

같은 사건을 다루는 기사들에 대해 LLM이 매번 조금씩 다른 사건명을 생성하여 불필요한 Case Group이 폭증했습니다. (예: "쿠팡 물류 사고"와 "쿠팡 배송 노동자 사망"이 별개 그룹으로 생성)

해결

2중 안전장치를 구현했습니다. ① 프롬프트에 기존 사건 목록(최근 90일, 최대 150건)을 주입하여 LLM이 기존 이름을 재사용하도록 유도. ② Python SequenceMatcher + 핵심 키워드 매칭으로 AI가 새 이름을 만들어도 기존 그룹에 귀속시킵니다.

문제 3

수집/분석 주기 충돌

수집(60분)과 분석(5분)이 하나의 파이프라인 잡으로 묶여 있어, 분석이 오래 걸리면 수집이 지연되고, 분석 중에 새 수집을 시작하면 DB 트랜잭션이 꼬이는 문제가 있었습니다.

해결

APScheduler의 두 개 독립 잡(crawl, analyze)으로 분리하고, 각각 독립 락(threading.Lock)을 사용했습니다. 잡 완료 후 다음 실행을 동적으로 재예약하여 오버랩을 원천 차단했습니다.

문제 4

적합도 분포 N+1 쿼리

분석 목록의 케이스 단위 뷰에서 각 케이스별 High/Medium/Low 기사 수를 개별 쿼리로 조회하면 케이스 수만큼 쿼리가 발생했습니다.

해결

ViewSet의 get_queryset()에 Django ORM Count + filter=Q(...) 어노테이션을 추가하여 단일 SQL 쿼리에서 분포를 계산했습니다. suitability_distribution SerializerMethodField로 프론트엔드에 깔끔하게 노출했습니다.

사건 그룹핑 핵심 로직

# 법률 불용어 제거 후 핵심 엔티티 추출
LAW_STOPWORDS = {"소송", "피해", "손해", "배상", "집단", "공동", "민원", ...}

def _case_similarity(name_a: str, name_b: str) -> float:
    # 1단계: 기본 문자열 유사도
    base_ratio = SequenceMatcher(None, name_a, name_b).ratio()

    # 2단계: 핵심 키워드(불용어 제거 후) 겹침 보너스
    kw_a = {w for w in name_a.split() if w not in LAW_STOPWORDS}
    kw_b = {w for w in name_b.split() if w not in LAW_STOPWORDS}
    overlap = len(kw_a & kw_b)

    bonus = min(overlap * 0.1, 0.3)
    return min(base_ratio + bonus, 1.0)

# 3단계 매칭 순서
# ① 정확 이름 일치 → 기존 그룹
# ② similarity ≥ 0.85 → 기존 그룹
# ③ 제목 키워드 4개 이상 공유 → 기존 그룹
# ④ 신규 CaseGroup 생성 (2026-MM-XXX)
LLM Design

LLM 프롬프트 설계

단순 요약 요청이 아닌, 소송금융 심사역의 판단 기준을 코드화한 구조화된 프롬프트입니다.

판단 기준 내용 등급 산출
C1상대방의 법적 책임이 명확함 충족 4개↑ → High
충족 2~3개 → Medium
충족 1개↓ → Low

X1 해당 시 → Low
C2상대방 자력이 충분함 (대기업, 상장사 등)
C3집단 피해 존재 (수십 명 이상)
C4피해 규모가 큼 (수억 원 이상)
C5증거 확보 가능
C6공적 절차 진행 중
X1이미 종결된 사건 (판결 확정, 합의 완료)—
"AI에게 단순히 '중요한 뉴스인지 판단해줘'라고 하지 않고, 소송금융 심사역이 사용하는 실제 기준(C1~C6)을 프롬프트에 정의하고 Few-shot 예시 3건을 함께 제공했습니다. temperature=0.1로 설정하여 판정의 일관성을 확보했습니다."

Design Decisions

주요 설계 결정

각 기술 선택에는 명확한 이유가 있습니다.

결정 선택 이유
스케줄링 APScheduler Celery+Redis 대신 단일 프로세스 내 실행. Redis 서버 불필요. 내부 도구에 적합한 단순 구조.
DB PostgreSQL (Docker) SQLite 동시 쓰기 잠금 문제 해결. 운영 환경과 동일한 DB를 개발 단계부터 사용.
LLM Gemini 2.5 Flash 무료 티어에서 충분한 성능. 긴 프롬프트(Few-shot + 기사 본문) 처리에 유리한 컨텍스트 윈도우.
프론트엔드 Vite + React 내부 업무 도구 특성상 빠른 개발 속도 우선. TypeScript로 API 응답 타입 안정성 확보.
패키지 관리 uv + bun pip 대비 10~100배 빠른 설치 속도. lockfile 기반 재현 가능한 환경.
API 설계 group_by_case 파라미터 기사 단위 / 케이스 단위 뷰를 단일 엔드포인트에서 처리. 프론트엔드 API 호출 단순화.
