# PRD — 법률 분쟁 사건 자동 발굴 시스템

> **Product Requirements Document v1.2**
> 최종 수정: 2026-03-05

---

## 1. 제품 개요

### 1-1. 한 줄 요약

법률 뉴스를 자동 수집하고 AI가 소송금융 투자 적합도를 판정하여 대시보드로 보여주는 시스템

### 1-2. 배경

로앤굿(LawNGood)은 소송금융 전문 법무법인입니다.
소송금융이란 승소 가능성이 높은 소송에 자금을 투자하고, 승소 시 수익을 얻는 비즈니스입니다.

현재 투자 대상 사건을 발굴하는 방식:

| 항목 | 현재 (수동) | 목표 (자동화) |
|------|------------|-------------|
| 뉴스 모니터링 | 담당자가 매일 포털에서 검색 | 네이버 뉴스 API로 자동 수집 |
| 적합성 판단 | 담당자가 기사를 읽고 주관적 판단 | AI가 6가지 기준(C1~C6)으로 객관적 판정 |
| 사건 정리 | 엑셀에 수작업 분류 | AI가 자동 사건 그룹핑 |
| 보고서 작성 | 엑셀 수작업 | 대시보드 + 엑셀 자동 내보내기 |

### 1-3. 목표 사용자

- **1차 사용자**: 로앤굿 소송금융팀 (사건 발굴 담당자)
- **2차 사용자**: 법무법인 대표 / 의사결정권자 (대시보드 확인)

### 1-4. 핵심 가치

1. **시간 절약**: 매일 2~3시간 소요되던 뉴스 모니터링을 5분으로 단축
2. **누락 방지**: 7개 키워드 x 100건 = 최대 700건을 빠짐없이 수집
3. **객관적 판단**: AI의 일관된 기준으로 적합도 판정 (사람의 주관성 제거)
4. **사건 추적**: 동일 사건에 대한 여러 기사를 자동으로 묶어서 추적 가능

## 3. 기능 요구사항

### 3-1. 뉴스 수집 (Crawling)

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| CR-01 | 네이버 뉴스 검색 API로 활성 키워드별 최대 100건 수집 | P0 |
| CR-02 | URL 기반 중복 기사 자동 필터링 | P0 |
| CR-03 | 네이버 뉴스 페이지에서 기사 본문 자동 추출 | P0 |
| CR-04 | URL 도메인 매핑(60개+) 또는 네이버 페이지 스크래핑으로 언론사 자동 식별 | P1 |
| CR-05 | 수집된 기사는 `pending` 상태로 저장 | P0 |
| CR-06 | 웹 UI 설정 페이지에서 키워드 추가/삭제 가능 | P1 |

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| AN-01 | `pending` 상태 기사를 순차적으로 LLM 호출하여 분석 | P0 |
| AN-02 | 소송금융 적합도를 High / Medium / Low로 판정 | P0 |
| AN-03 | 판정 근거를 C1~C6, X1 기준 코드로 설명 | P0 |

## 4. AI 분석 판정 기준

### 적합 조건 (C1 ~ C6)

| 코드 | 기준 | 설명 |
|------|------|------|
| C1 | 상대방 책임 명확 | 법적 의무 위반, 과실 등이 인정되는 경우 |
| C2 | 상대방 자력 충분 | 대기업, 상장사, 정부기관 등 배상 능력이 있는 경우 |
| C3 | 집단적 피해 존재 | 피해자가 수십 명 이상인 경우 |
| C4 | 피해 규모가 큼 | 수억 원 이상 또는 피해자 수만 명 이상 |
| C5 | 증거 확보 가능 | 증거가 있거나 확보할 수 있는 경우 |
| C6 | 공적 절차 진행 중 | 수사, 감사, 행정조치 등 (소송 제외) |

### 부적합 조건 (X1)

| 코드 | 기준 | 설명 |
|------|------|------|
| X1 | 이미 종결된 사건 | 판결 확정, 합의 완료 등 |

### 등급 산정

| 등급 | 조건 |
|------|------|
| **High** | 적합 조건 4개 이상 충족, 부적합 조건 없음 |
| **Medium** | 적합 조건 2~3개 충족, 부적합 조건 없음 |
| **Low** | 적합 조건 1개 이하, 또는 부적합 조건 해당 |

## 5. 데이터 파이프라인

```
 ① 수집 (just crawl)          ② 분석 (just analyze)         ③ 결과 확인
 ─────────────────           ─────────────────           ─────────────

 키워드 7개 순회               pending 기사 순회             웹 대시보드
      │                           │                          │
      ▼                           ▼                          ▼
 네이버 API 호출              Gemini API 호출            통계/차트 확인
 (키워드당 100건)             (기사당 ~4초)              필터/검색
      │                           │                      엑셀 내보내기
      ▼                           ▼
 중복 제거 + 본문 추출        JSON 파싱 + 검증
      │                           │
      ▼                           ▼
 Article 저장               Analysis 저장
 status=pending             사건 그룹 자동 연결
                            status=analyzed
```

### 파이프라인 실행 방법

| 명령어 | 설명 |
|--------|------|
| `just crawl` | 뉴스 수집만 실행 |
| `just analyze` | AI 분석만 실행 |
| `just pipeline` | 수집 → 분석 전체 실행 |
| `just reanalyze <ID>` | 특정 기사 재분석 |

## 9. 주요 기술 결정 변경 이력

> 초기 설계와 실제 구현 사이에 발생한 결정 변경과 그 이유를 기록합니다.
> 이력서/포트폴리오에서 "문제 인식 → 원인 분석 → 해결" 근거로 활용 가능합니다.

---

### 결정 1. DB: SQLite → PostgreSQL

| 항목 | 내용 |
|------|------|
| **초기 설계** | SQLite (로컬 단일 파일 DB, 의존성 없음) |
| **발생 문제** | APScheduler가 백그라운드 스레드에서 AI 분석을 실행하는 동안, Django 개발 서버의 다른 요청이 DB에 쓰기를 시도하면 `OperationalError: database is locked` 반복 발생 |
| **원인** | SQLite는 동시 쓰기를 지원하지 않음. 단일 쓰기 락 때문에 수집 스레드와 웹 요청 스레드가 충돌 |
| **해결** | PostgreSQL 16 (Docker)으로 마이그레이션. Django `settings.py`에 `DATABASE_URL` 파싱 로직 직접 구현 (추가 라이브러리 없이 전환). 기존 SQLite 데이터는 `serializers.serialize/deserialize`로 마이그레이션 |
| **교훈** | 멀티스레드 환경에서는 처음부터 PostgreSQL을 사용해야 한다. "나중에 전환"은 마이그레이션 비용이 생긴다 |

---

### 결정 2. 스케줄링: Celery + Redis → APScheduler

| 항목 | 내용 |
|------|------|
| **초기 설계** | PRD v1.0에 Celery + Redis 비동기 태스크 큐 명시 (v2.0 예정 항목) |
| **결정 변경** | APScheduler (Django 프로세스 내 실행) 채택 |
| **이유** | 내부 업무 도구 특성상 인프라 복잡도 최소화 우선. Redis 서버를 별도 운영할 필요 없이 APScheduler로 수집(60분) + 분석(5분) 자동화가 충분히 가능. Celery는 대규모 분산 처리가 필요할 때 도입 검토 |
| **트레이드오프** | 프로세스 재시작 시 잡 스케줄 리셋. 다중 워커 운영 불가. 현재 요구사항에서는 허용 가능한 제약 |

---

### 결정 3. 스케줄러 구조: 단일 파이프라인 잡 → 두 개 독립 잡

| 항목 | 내용 |
|------|------|
| **초기 구현** | 수집과 분석을 `_run_pipeline_job()` 하나로 묶어 60분마다 실행 (수집 완료 후 곧바로 분석 실행) |
| **발생 문제** | 분석이 오래 걸리면 다음 수집이 지연됨. 분석 중 새 수집이 시작되면 DB 쓰기 충돌 가능. 두 작업의 주기가 달라야 하는 요구사항 반영 불가 (수집은 60분, 분석은 5분이 적합) |
| **해결** | `crawl` 잡과 `analyze` 잡을 APScheduler에 각각 독립 등록. 각 잡은 `threading.Lock`으로 동시 실행 방지. 잡 완료 후 다음 실행을 동적으로 재등록 (DateTrigger one-shot 방식) |
| **교훈** | 수집 주기와 분석 주기는 본질적으로 다르다. 처음부터 분리 설계하는 것이 맞다 |
