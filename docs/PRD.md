# PRD — 법률 분쟁 사건 자동 발굴 시스템

> **Product Requirements Document v1.0**
> 최종 수정: 2026-02-27

---

## 1. 제품 개요

### 1-1. 한 줄 요약

법률 뉴스를 자동 수집하고 AI가 소송금융 투자 적합도를 판정하여 대시보드로 보여주는 시스템

### 1-2. 배경

로앤굿(LawNGood)은 소송금융 전문 법무법인입니다.
소송금융이란 승소 가능성이 높은 소송에 자금을 투자하고, 승소 시 수익을 얻는 비즈니스입니다.

현재 투자 대상 사건을 발굴하는 방식:

| 항목 | 현재 (수동) | 목표 (자동화) |
|------|------------|-------------|
| 뉴스 모니터링 | 담당자가 매일 포털에서 검색 | 네이버 뉴스 API로 자동 수집 |
| 적합성 판단 | 담당자가 기사를 읽고 주관적 판단 | AI가 6가지 기준(C1~C6)으로 객관적 판정 |
| 사건 정리 | 엑셀에 수작업 분류 | AI가 자동 사건 그룹핑 |
| 보고서 작성 | 엑셀 수작업 | 대시보드 + 엑셀 자동 내보내기 |

### 1-3. 목표 사용자

- **1차 사용자**: 로앤굿 소송금융팀 (사건 발굴 담당자)
- **2차 사용자**: 법무법인 대표 / 의사결정권자 (대시보드 확인)

### 1-4. 핵심 가치

1. **시간 절약**: 매일 2~3시간 소요되던 뉴스 모니터링을 5분으로 단축
2. **누락 방지**: 7개 키워드 x 100건 = 최대 700건을 빠짐없이 수집
3. **객관적 판단**: AI의 일관된 기준으로 적합도 판정 (사람의 주관성 제거)
4. **사건 추적**: 동일 사건에 대한 여러 기사를 자동으로 묶어서 추적 가능

## 3. 기능 요구사항

### 3-1. 뉴스 수집 (Crawling)

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| CR-01 | 네이버 뉴스 검색 API로 활성 키워드별 최대 100건 수집 | P0 |
| CR-02 | URL 기반 중복 기사 자동 필터링 | P0 |
| CR-03 | 네이버 뉴스 페이지에서 기사 본문 자동 추출 | P0 |
| CR-04 | URL 도메인 매핑(60개+) 또는 네이버 페이지 스크래핑으로 언론사 자동 식별 | P1 |
| CR-05 | 수집된 기사는 `pending` 상태로 저장 | P0 |
| CR-06 | 웹 UI 설정 페이지에서 키워드 추가/삭제 가능 | P1 |

### 3-2. AI 분석 (Analysis)

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| AN-01 | `pending` 상태 기사를 순차적으로 LLM 호출하여 분석 | P0 |
| AN-02 | 소송금융 적합도를 High / Medium / Low로 판정 | P0 |
| AN-03 | 판정 근거를 C1~C6, X1 기준 코드로 설명 | P0 |
| AN-04 | 사건 분야, 상대방, 피해 규모, 피해자 수, 소송 단계 추출 | P0 |
| AN-05 | 3~5문장 요약 생성 | P0 |
| AN-06 | 법적 분쟁과 무관한 기사를 `is_relevant=false`로 자동 필터링 | P1 |
| AN-07 | 기존 사건 그룹 목록을 프롬프트에 주입하여 동일 사건명 재사용 유도 | P1 |
| AN-08 | AI가 새 사건명을 생성한 경우 유사도 매칭(0.6 이상)으로 기존 그룹에 자동 연결 | P1 |
| AN-09 | Gemini 기본, OpenAI 폴백 전략 | P0 |
| AN-10 | 분석 실패 시 `failed` 상태 저장, 재분석 기능 제공 | P1 |

### 3-3. 사건 그룹핑 (Case Grouping)

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| CG-01 | AI가 생성한 사건명(case_name)으로 CaseGroup 자동 생성 | P0 |
| CG-02 | CASE-YYYY-NNN 형식의 고유 사건 ID 자동 부여 | P1 |
| CG-03 | 정확 일치 → 유사도 매칭 → 신규 생성 3단계 매칭 | P1 |
| CG-04 | 법률 일반 용어(소송, 피해, 분쟁 등) 제외한 핵심 엔티티 기반 유사도 계산 | P2 |

### 3-4. 대시보드 & 프론트엔드

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| FE-01 | 대시보드: 오늘 수집 건수, 분석 완료 건수, High/Medium 건수 통계 카드 | P0 |
| FE-02 | 대시보드: 적합도 분포 도넛 차트, 사건 분야 바 차트, 주간 추이 라인 차트 | P1 |
| FE-03 | 분석 목록: 적합도/단계/날짜/검색어 필터링 + 페이지네이션 | P0 |
| FE-04 | 분석 목록: 사건별 그룹핑 보기 (동일 사건은 대표 1건 + "+N" 배지) | P1 |
| FE-05 | 분석 목록: 무관 기사 포함/제외 토글 | P1 |
| FE-06 | 분석 상세: AI 요약, 판단 근거, 사건 정보, 유사 기사 목록 | P0 |
| FE-07 | 분석 상세: 재분석 요청 버튼 | P1 |
| FE-08 | 엑셀 다운로드 (분석 목록에서 현재 필터 조건 기반) | P1 |
| FE-09 | 설정 페이지: 수집 키워드 관리 (추가/삭제) | P1 |

## 4. AI 분석 판정 기준

### 적합 조건 (C1 ~ C6)

| 코드 | 기준 | 설명 |
|------|------|------|
| C1 | 상대방 책임 명확 | 법적 의무 위반, 과실 등이 인정되는 경우 |
| C2 | 상대방 자력 충분 | 대기업, 상장사, 정부기관 등 배상 능력이 있는 경우 |
| C3 | 집단적 피해 존재 | 피해자가 수십 명 이상인 경우 |
| C4 | 피해 규모가 큼 | 수억 원 이상 또는 피해자 수만 명 이상 |
| C5 | 증거 확보 가능 | 증거가 있거나 확보할 수 있는 경우 |
| C6 | 공적 절차 진행 중 | 수사, 감사, 행정조치 등 (소송 제외) |

### 부적합 조건 (X1)

| 코드 | 기준 | 설명 |
|------|------|------|
| X1 | 이미 종결된 사건 | 판결 확정, 합의 완료 등 |

### 등급 산정

| 등급 | 조건 |
|------|------|
| **High** | 적합 조건 4개 이상 충족, 부적합 조건 없음 |
| **Medium** | 적합 조건 2~3개 충족, 부적합 조건 없음 |
| **Low** | 적합 조건 1개 이하, 또는 부적합 조건 해당 |

## 5. 데이터 파이프라인

```
 ① 수집 (just crawl)          ② 분석 (just analyze)         ③ 결과 확인
 ─────────────────           ─────────────────           ─────────────

 키워드 7개 순회               pending 기사 순회             웹 대시보드
      │                           │                          │
      ▼                           ▼                          ▼
 네이버 API 호출              Gemini API 호출            통계/차트 확인
 (키워드당 100건)             (기사당 ~4초)              필터/검색
      │                           │                      엑셀 내보내기
      ▼                           ▼
 중복 제거 + 본문 추출        JSON 파싱 + 검증
      │                           │
      ▼                           ▼
 Article 저장               Analysis 저장
 status=pending             사건 그룹 자동 연결
                            status=analyzed
```

### 파이프라인 실행 방법

| 명령어 | 설명 |
|--------|------|
| `just crawl` | 뉴스 수집만 실행 |
| `just analyze` | AI 분석만 실행 |
| `just pipeline` | 수집 → 분석 전체 실행 |
| `just reanalyze <ID>` | 특정 기사 재분석 |

## 6. 기술 스택

### 백엔드

| 기술 | 역할 |
|------|------|
| Python 3.12+ | 메인 언어 |
| Django 5.x + DRF | 웹 프레임워크 + REST API |
| Gemini 2.5 Flash | 기본 LLM (AI 분석) |

## 7. REST API 엔드포인트

| Method | URL | 설명 |
|--------|-----|------|
| GET | `/api/analyses/` | 분석 결과 목록 (필터, 검색, 페이지네이션) |
| GET | `/api/analyses/{id}/` | 분석 결과 상세 |
| GET | `/api/analyses/stats/` | 대시보드 통계 |
| GET | `/api/analyses/export/` | 엑셀 다운로드 |
| GET | `/api/case-groups/` | 사건 그룹 목록 |
| GET | `/api/case-groups/{id}/` | 사건 그룹 상세 |
| GET | `/api/articles/` | 기사 목록 |
| GET | `/api/articles/{id}/` | 기사 상세 |
| POST | `/api/articles/{id}/reanalyze/` | 기사 재분석 |
| GET | `/api/keywords/` | 키워드 목록 |
| POST | `/api/keywords/` | 키워드 추가 |
| DELETE | `/api/keywords/{id}/` | 키워드 삭제 |
| GET | `/api/sources/` | 언론사 목록 |
| GET | `/api/docs/` | Swagger API 문서 |
